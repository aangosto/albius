# Albius — contexto del proyecto

> Este fichero es la fuente única de reglas y decisiones del proyecto. Cualquier divergencia entre lo que digo aquí y otro sitio se resuelve a favor de este documento.

## 1. Qué es

SaaS B2B multi-tenant para gestión de turnos en empresas de transporte urbano de viajeros.
Target: empresas medianas (50-300 conductores).
Roles: super_admin (autor del producto), jefe de tráfico (operativa diaria), conductor (móvil-first).

## 2. Diferenciadores

1. **Optimizador matemático** que respeta el convenio y maximiza preferencias individuales del conductor.
2. **Mercado oficial de intercambios** entre conductores con validación automática y aprobación del jefe.

## 3. Stack

- Frontend: React 19 + Vite 6 + TypeScript strict + Tailwind v4 + shadcn/ui (estilo `new-york`, base `slate`).
- Backend: Firebase (Auth + Firestore + Cloud Functions Node).
- Optimizador: **Python + Pyomo + HiGHS** (validado en spike a escala TUCARSA; **OR-Tools descartado**) en Cloud Run. Motor MILP de assignment plaza↔conductor; servicio FastAPI en `services/optimizer/` (B29 Fase A). Su DEMANDA se deriva de `TipoTurno.tiposDiaAplicables` + `resolverTipoDia` (calendario, B27); el optimizador Python reimplementa `resolverTipoDia` (el contrato compartido es la colección `festivos`, no la función TS).
- Despliegue: Vercel con CI/CD automático desde `main`.

## 8. Decisiones tomadas (no cuestionar sin avisar)

Si propones algo que contradice un punto de aquí, primero levanta la mano para abrir conversación.

- **Aislamiento multi-tenant con campo `tenantId` en cada documento Firestore** (colecciones planas), no subcolecciones anidadas bajo cada tenant.
- **Servicios materializados selectivamente:** solo se crea documento `servicios` cuando hay asignación o incidencia que registrar. El resto se calcula al vuelo desde `frecuencias`.
- **Intercambios solo inmediatos en MVP:** únicamente intercambios sobre fechas ya publicadas. Los intercambios diferidos (compromisos futuros entre conductores) quedan fuera.
- **Horizonte del cuadrante: 2 meses publicados.** Los intercambios solo son posibles dentro de ese horizonte.
- **Optimizador en dos fases:** heurística greedy con scoring en MVP (suficiente hasta ~200 conductores); formulación MILP con OR-Tools en V2. Interfaz estable para cambiar de motor sin tocar el resto del sistema.
  - **NOTA DE ACTUALIZACIÓN (B23, revisar al diseñar el bloque Optimizador):** (a) **La DEMANDA del optimizador en MVP sale de los TIPOS DE TURNO activos por día, NO de las frecuencias.** Las `frecuencias` (B23) son dato **descriptivo** del servicio (intensidad de paso); el puente frecuencias→expediciones→turnos (vehicle + crew scheduling) queda fuera del MVP. El "servicios materializados desde frecuencias" del bullet anterior es el cálculo de expediciones para visualización/operativa, NO el insumo directo del optimizador de turnos. (b) **Un spike Pyomo validó que MILP directo (assignment plaza↔conductor con preferencias) es viable a escala real (~3-4 min con 178 conductores)**, así que el optimizador MVP podría ser **MILP directo en vez de la heurística greedy** que este bullet preveía — reevaluar greedy-vs-MILP-directo cuando se diseñe el bloque, con la interfaz estable de motor intacta. (c) **El `Convenio` (B25) define POR CENTRO los parámetros que el optimizador leerá como restricciones** (descanso entre jornadas, max horas semanales/anuales, días libres, etc.); el optimizador resolverá el `maxHoras` efectivo de cada conductor como `conductor.maxHorasSemanales ?? convenio.maxHorasSemanales` (el convenio pone el límite del centro, el conductor lo sobreescribe individualmente). (d) **NOTA B26 (la "mesa" ya existe):** el optimizador escribirá las asignaciones que decida vía `crearAsignacionesLote` sobre un `Cuadrante` en estado `borrador`, y los KPIs del resultado (cobertura, satisfacción, preferencias cumplidas) en `Cuadrante.estadisticas` vía `actualizarCuadrante` (callable aún por implementar — `TODO[actualizar-cuadrante-estadisticas]`). **Grano mensual:** un `Cuadrante` cubre un centro × un mes (id `cua_{centroId}_{año}_{mes}`); el horizonte de 2 meses publicados se materializa como 2 documentos `Cuadrante`. El descanso ordinario del conductor NO se materializa (ausencia de asignación = día libre, D6.10). (e) **NOTA B27 (la DEMANDA y el CALENDARIO ya existen como dato):** la demanda del optimizador para un día concreto = los **tipos de turno activos del centro cuyo `tiposDiaAplicables` incluye el `tipoDia` de ese día** (1 plaza por tipo y día; fiel a TUCARSA donde cada código de turno aparece máx. 1 vez/día). El `tipoDia` de cada fecha se resuelve con `resolverTipoDia(fecha, festivosAplicables)` (`@albius/shared/calendar`, función pura: el festivo gana sobre el día de semana; el caller pre-filtra los festivos del centro). La colección `festivos` ya está habilitada (CRUD por callables; **frontend pendiente**, `TODO[festivos-frontend]` — al inicio se siembran por seed). (f) **NOTA B28 (la escritura del lado Node está LISTA):** el optimizador (B29) llamará, desde un orquestador Node, a `regenerarAsignaciones` (limpiar+volcar el plan) y a `actualizarCuadranteDoc` (KPIs en `estadisticas` + bloque de generación `generadoPor`/`fechaGeneracion`/`modoGeneracion`) como **funciones internas** de `apps/functions/src/logic/` (patrón D6.12: escritura reutilizable sin `request.auth`), con `actorId='optimizador'`. Patrón de invocación previsto **C1** (la Function lee Firestore → pasa el payload al Python → recibe el plan → escribe; el Python NO toca Firestore), a cerrar en B29. Invocar Cloud Run requerirá `google-auth-library` (dep nueva) + decisión async (el spike tarda 3-4 min > timeout de un `onCall`). Lo único que falta para el optimizador es el motor (Python/OR-Tools/Cloud Run) y el orquestador que pegue lectura→Python→escritura.
- **Una única aplicación web responsive** sirve a los tres roles desde móvil y escritorio. Sin app nativa separada en V1; en V2 se podría empaquetar con React Native reutilizando código.
- **Acceso multidispositivo para todos los roles:** super_admin, jefe de tráfico y conductor acceden a la misma aplicación web responsive desde cualquier dispositivo (móvil o escritorio), con las mismas capacidades funcionales.
- **Sin mapa interactivo en MVP:** las paradas se gestionan con texto y coordenadas opcionales. Visualización cartográfica queda para V2.

### Bloque 3 — callables crearJefeTrafico + crearConductor

- **D3.1 — `conductorId` desacoplado del `uid` de Firebase Auth.** El identificador del conductor coincide con el número de empleado (información de negocio que debe sobrevivir a cambios de cuenta de Auth) y NO con el `uid` del documento `/usuarios`. El enlace usuario↔conductor se hace mediante el campo `conductorId` en el documento `/usuarios`.
- **D3.2 — `crearConductor` crea `/usuarios` y `/conductores` atómicamente.** El callable escribe ambos documentos en una única operación. Si alguna de las dos escrituras falla, rollback completo (incluyendo el usuario de Firebase Auth si ya se creó). Un conductor sin sus dos documentos no es funcional.
- **D3.3 — Contraseña inicial vía `generatePasswordResetLink`.** El email de configuración de contraseña se envía con `generatePasswordResetLink` del Admin SDK. NUNCA devolver contraseñas en la respuesta del callable ni escribirlas en logs. Si el usuario pierde el enlace, se repite el flujo desde super_admin.
- **D3.4 — Validación de payloads con type guards a mano.** En este bloque NO se introduce Zod: se valida cada campo con type guards y se lanza `invalid-argument` con mensaje claro al fallar. Si los callables crecen a 10 o más se refactoriza (ver `TODO[refactor-zod]` en §12).
- **D3.5 — Sesión 3 partida en dos sub-sesiones por tamaño.** Bloque 3.1: scaffold del paquete `apps/functions` + callable `ping` (completado). Bloque 3.2: callables `crearJefeTrafico` y `crearConductor`.
- **D3.6 — Verificación de existencia de referencias antes de crear.** `crearJefeTrafico` verifica que `/tenants/{tenantId}` y `/centros/{centroId}` existen. `crearConductor` verifica lo mismo y además que el `tenantId` del payload coincide con el `tenantId` del jefe que llama (anti cross-tenant). Si falla la verificación, devolver `invalid-argument` indicando qué referencia no existe.
  - **Ampliación 3.2.d:** para `crearConductor` se verifica también que `claims.centroId === payload.centroId` cuando invoca un jefe (anti cross-centro), por simetría con la identidad operativa del jefe `(tenantId, centroId)`. Mensaje: "Un jefe de tráfico no puede crear conductores en otro centro." (DUDA-11 de 3.2.d). Sin esta ampliación, un jefe del centro A podría crear conductores en centro B del mismo tenant, lo cual es abuso de permisos. Implementación: commit `e810832`.
- **D3.7 — Auditoría mínima en cada documento creado.** Los documentos nuevos en `/usuarios` y `/conductores` incluyen `creadoPor` (uid del invocador, `request.auth.uid`) y `creadoEn` (`FieldValue.serverTimestamp()`).
  - **Ampliación Bloque 4:** bootstrap CLI usa `creadoPor: 'bootstrap-cli'` como valor convencional al no existir `request.auth.uid` (script ejecutado fuera del flujo callable, sin invocador autenticado). Implementación: `scripts/bootstrap-super-admin.mjs`. Grep-eable para localizar todos los super_admins creados por bootstrap CLI vs por callables (que tendrían un uid real en `creadoPor`).

### Bloque 6 — routing protegido + sidebar por rol

- **D6.1 — ProtectedRoute solo gatea autenticación, no rol específico de ruta.** Defensa en profundidad por capas: UI (sidebar oculta items que el rol no debe ver, mejora UX), routing (`<ProtectedRoute/>` valida status+user.rol, evita accidentes con URL bar), backend (reglas Firestore rechazarán lecturas cuando lleguen en sesiones 4+, ÚNICA capa de defensa real). Si surge necesidad de `RoleGate` explícito por ruta, sub-bloque aparte sin tocar este contrato.
- **D6.5 — `homeForRol(rol)` agnóstico al destino.** Mapping `ROL_HOME: Record<Rol, string>` desacopla la convención. Añadir un cuarto rol futuro (ej. `inspector`) requiere solo añadir entrada al record, sin tocar `LoginPage`, `ProtectedRoute` ni `NotFoundPage`.

### Bloque 7 — forzado de cambio de password en primer login

- **D7.3 — `refreshAuthUser()` en AuthContext re-lee el doc `/usuarios` y actualiza `user` state.** Establece patrón canónico: cualquier mutación server-side que el frontend deba reflejar en estado inmediato se canaliza por este método. Aplicable a hidrataciones futuras (tenant/centro en Topbar — ver `TODO[topbar-tenant-centro-hidratado]`) y a estados que dependan del doc `/usuarios`.

Tocar la regla `read` self-only sobre `/usuarios` rompe el flujo de login del Bloque 7 (hay un comentario explicativo encima de la regla en `firestore.rules`).

### Bloque 4 — Tenants y Centros

- **D4.1 — `actualizadoPor` + `actualizadoEn` como auditoría obligatoria de UPDATE en todo callable de modificación.** Refinamiento de D3.7 (que solo cubría CREATE con `creadoPor`/`creadoEn`). Todo callable cuyo verbo es "actualizar" escribe ambos campos en cada write, sin excepción. Implicación: los tipos del modelo añaden `actualizadoPor?: string` y `actualizadoEn?: Timestamp` (opcionales para retrocompatibilidad con docs legados). Aplica retrospectivamente a futuras refactorizaciones de `marcarPasswordCambiada` si se considera UPDATE auditable.
- **D4.2 — Defaults de campos opcionales aplicados en backend, no en frontend.** Los callables de creación rellenan defaults (`configuracion.zonaHoraria='Europe/Madrid'`, `configuracion.idioma='es'`, `estado='activo'`, `plan='basico'`, etc.) cuando el payload los omite. El form puede no enviarlos. Centraliza la regla en un único lugar y evita drift cliente/servidor. Aplica retroactivamente a `crearJefeTrafico`, `crearConductor` y `bootstrap-super-admin.mjs`, que ya defaultean campos del modelo (`passwordChangeRequired=true`, `estado='activo'`, etc.) en backend; D4.2 formaliza el patrón implícito.
- **D4.3 — Soft-delete se implementa como UPDATE del callable `actualizar*` con cambio del campo `estado`, NUNCA callable separado `eliminar*`.** Hard-delete reservado a super_admin manual desde consola Firebase para limpieza de datos contaminados (sin UI). Las reglas Firestore permiten `delete` solo a super_admin pero la web NO expone esa acción. Implicación: simplifica el menú de callables (4 por entidad principal: crear, actualizar, …) y unifica auditoría (`actualizadoPor` registra quién canceló/inactivó).
- **D4.4 — Validaciones de dominio (CIF, futuras: DNI, IBAN, matrículas) viven en `packages/shared/src/validators/` como funciones puras sin dependencias externas, compartidas entre frontend y backend.** Mientras `TODO[refactor-shared-build]` esté abierto, el validador se duplica SOLO en `apps/functions/src/cif-validator.ts` (copia necesaria porque `tsc → CJS` no resuelve los .ts de shared en runtime). `apps/web` consume la copia canónica de `packages/shared` directamente vía `@albius/shared` (Vite resuelve los .ts crudos del paquete). La copia 3 anticipada en B8 (`apps/web/src/lib/validators/cif.ts`) resultó redundante — ver `TODO[cif-copia3-redundante]`. La validación se ejecuta en frontend (UX instant feedback en blur) Y en backend (gate).

  Para CIF específicamente, el gate del backend admite escape hatch (`forzarCIF: true`) para casos legítimos no cubiertos por el algoritmo español estándar (empresas extranjeras, autónomos con DNI, sociedades civiles): el operador confirma conscientemente y se persiste flag `cifValidacionForzada: true` para auditoría. El patrón "escape hatch con flag de auditoría" será reusable para futuros validators de dominio cuando el caso real lo justifique.

- **D4.5 — Objetos compuestos en payload de callables (`configuracion`, futuros similares) tienen tratamiento asimétrico CREATE vs UPDATE.** CREATE: merge del objeto recibido con defaults backend. Permite payload parcial, los huecos se rellenan automáticamente. UPDATE: replace completo. El validator exige que el objeto incluya todos los sub-campos del modelo. Si el frontend quiere editar un sub-campo, debe hidratar primero el objeto entero, modificarlo, y re-enviarlo completo.

  Razón: evita ambigüedad de "envío `{idioma: 'en'}`, ¿qué pasa con `zonaHoraria`?". El replace en UPDATE hace explícita la responsabilidad del frontend. La asimetría es intencional: el alta es "rellena lo que falta", la edición es "este es el estado que quiero".

- **D4.6 — Soft-delete (cambio de estado a `'cancelado'`/`'inactivo'`) verifica que no hay entidades hijas activas dependientes antes de aceptar el cambio.** Si las hay, el callable rechaza con `failed-precondition` indicando qué bloquea (con conteo o lista). Cascada manual: el operador debe inactivar primero las dependencias. NO se implementa cascada automática (riesgo de borrado masivo accidental). Implementaciones del Bloque 9: Tenant → verifica Centros activos via query `where(tenantId, ==, X).where(estado, ==, 'activo')`. Aplicable simétricamente a Centro → Conductores activos (futuro Sesión 7).

- **D4.7 — Modal Dialog para CRUDs de entidades de Gobierno (Tenants, Centros, Usuarios).** Cuando el agente del CRUD es super_admin (uso interno, no comparte URLs), las altas y ediciones viven en Dialog modal con el listado de fondo, no en páginas separadas. Razón: preserva contexto + super_admins no comparten URLs. Reevaluar si una entidad pasa a tener 15+ campos. Origen: Bloque 10.

- **D4.8 — Listado client-side con re-fetch tras mutación, sin `onSnapshot`.** Para entidades administrativas de baja frecuencia (cardinality < 1000 y mutaciones < 10/día: Tenants/Centros/Usuarios en MVP), el frontend carga todo con `getDocs` sin paginación y filtra/busca client-side. Tras cada mutación exitosa, re-fetch completo del listado. NO usar `onSnapshot` (innecesario para datos administrativos de baja frecuencia, complica testing manual, consume reads). Paginación servidor cuando la entidad se acerque a 1000. Origen: Bloque 10.

- **D4.9 — Servicios `apps/web/src/lib/services/<entidad>.ts` como única superficie I/O Firebase desde la UI.** Componentes y páginas NO importan directamente `httpsCallable`/`getDocs`/`collection`. El servicio expone funciones tipadas que envuelven callables + queries. Centraliza tipos del wire, mapper de errores, transformaciones. Origen: Bloque 10 con `services/tenants.ts`.

- **D4.10 — Mapper de errores `HttpsCallable` centralizado (`lib/callable-errors.ts`).** Análogo a `auth-errors.ts`. Traduce códigos del wire (`invalid-argument`, `failed-precondition`, `unauthenticated`, `permission-denied`, `internal`, `unavailable`, `resource-exhausted`, `deadline-exceeded`) a mensajes en español. Preserva el mensaje específico del backend cuando es seguro mostrarlo (validation errors, mensajes user-friendly como el de D4.6); fallback genérico cuando no. Detección por duck typing, sin acoplar a la clase `FirebaseError`. Origen: Bloque 10.

- **D4.11 — Soft-delete UI vive en botón secundario destructivo separado del form (NO en el select de estado).** Form de edición excluye `'cancelado'` del select. Cancelación con Dialog de confirmación visualmente destructivo. Reactivación con botón "Reactivar" inline en fila de tenants cancelados. Refina D4.3 con su correspondiente decisión UX. Origen: Bloque 10.

- **D4.12 — Validators de dominio se ejecutan en `onBlur` con feedback inmediato + checkbox dinámico para escape hatch.** NO validar al cargar el form (no irritar al usuario antes de que escriba). Checkbox solo aparece cuando hay error real validable. Si el operador marca el checkbox, el form pasa `forzar*: true` al callable. Si edita el campo tras blur, se resetean tanto el flag de validado como el de forzar (el consentimiento se ata al valor concretamente evaluado). Aplicable a futuros validators (DNI, IBAN, matrículas) con la misma forma escape hatch. Origen: Bloque 10.

- **D4.13 — Páginas administrativas con gate suave de rol.** Las páginas reservadas a un rol específico (super_admin, jefe_trafico, etc.) renderizan `<NoAutorizadoView />` si `user.rol !== rolRequerido`. Defensa en profundidad sobre las reglas Firestore (que bloquean a nivel datos) + sobre la sidebar (que oculta el link). Evita render confuso si un user navega vía URL bar. **Patrón de implementación:** el gate vive en el componente exportado (ej. `TenantsPage`) que solo invoca `useAuth`; los hooks que dependen de permisos (`useState`/`useEffect`/`useMemo`) viven en un sub-componente (ej. `TenantsPageAuthorized`) renderizado solo si el gate pasa. Esto evita violación de Rules of Hooks si el rol cambia entre renders. Patrón reusable para `CentrosPage`, `UsuariosPage` y futuras pages administrativas. Origen: Bloque 10.

### Bloque 6 — Usuarios backend + frontend

- **D5.4 — Edición de campos dual-homed (Firestore + Firebase Auth) canalizada obligatoriamente por callable con escritura dual y rollback inverso.** Campos que viven a la vez en el doc Firestore y en el Auth user record (`email`→Auth email, `nombreCompleto`→Auth displayName) NUNCA se editan por escritura directa de cliente (las reglas `/usuarios` los vetan también para el owner). El callable escribe ambos sistemas en orden Auth primero, Firestore después: si la escritura Firestore falla, se revierte Auth al valor previo (rollback inverso, simétrico a `crear*`). El orden Auth-primero permite que colisiones (`email-already-exists`) aborten antes de tocar Firestore, sin rollback. Reusable para futuros campos dual-homed. Origen: Bloque 13.

  **Ampliación B21 — campos dual-homed ENTRE dos colecciones Firestore (no solo Firestore+Auth).** `crearConductor` copia `email`/`telefono`/`nombre`/`apellidos` tanto en `/usuarios` como en `/conductores`. Para evitar drift, el callable de la entidad SECUNDARIA veta esos campos y redirige al callable canónico de la entidad propietaria: `actualizarConductor` (B21) rechaza `email`/`telefono`/`nombre`/`apellidos` con `invalid-argument` y mensaje que apunta a `actualizarUsuario` (que sí mantiene la coherencia `/usuarios`↔Auth vía D5.4). El patrón "el campo se edita en UN solo callable, el resto lo vetan" se aplica tanto a Firestore+Auth (D5.4 original) como a colección↔colección.

- **D5.5 — Mutaciones de campos-identidad y claims se aíslan en callables dedicados, nunca en el `actualizar*` genérico.** `rol`, `tenantId`, `centroId` (custom claims) y `conductorId` (identidad del conductor, D3.1) quedan vetados en `actualizarUsuario` con `invalid-argument`. Sus cambios — cuando se necesiten — irán en callables dedicados con flags explícitos (`cambiarRolUsuario`, `moverUsuario`), porque arrastran re-set de claims + refresh de token en cliente + coherencia jerárquica. `conductorId` es inmutable de por vida. Mejor varios callables acotados que uno monstruoso. Origen: Bloque 13.

- **D5.6 — Callables que devuelven un secreto de un solo uso: el Dialog NO cierra al éxito; muestra pantalla intermedia con el secreto en input readonly + Copiar + aviso "no se mostrará de nuevo" + Cerrar explícito.** El secreto se renderiza antes del refetch del listado; el refetch va en background con `.catch` que solo loggea, para que un fallo de refetch nunca oculte el secreto. El cierre por overlay/Esc/X se bloquea mientras la pantalla de éxito está visible. Aplicable a `crearJefeTrafico`/`crearConductor` (devuelven `linkPasswordReset`, hoy distribuido manualmente — `TODO[email-transport]`) y a futuros callables con tokens/links de un solo uso. Origen: Bloque 14.

- **D5.7 — Selectores jerárquicos padre→hijo: dos `Select` encadenados donde el hijo se filtra por el padre seleccionado y se resetea al cambiar el padre.** El reset del hijo se encapsula en el componente del selector (no en cada form consumidor). Ambos niveles muestran solo entidades en estado operativo (activo). Si el padre no tiene hijos, el `Select` hijo se deshabilita con placeholder informativo y el submit queda bloqueado. Componente reutilizable (`TenantCentroSelect`). Aplicable a futuras jerarquías (centro→línea, línea→parada). Origen: Bloque 14.

### Bloque 7 — Líneas

- **D6.2 — Línea usa `estado` enum-3 (`'activa' | 'inactiva' | 'suspendida'`), excepción explícita y consciente a D5.3** (que mandaba binario activo/inactivo para entidades simples). Razón: casuística operativa más rica que Centro — la suspensión temporal estacional (línea de playa fuera de temporada) NO es lo mismo que la inactivación definitiva. Implica que el frontend de Líneas usa Select de estado (patrón Tenant D4.11), no botones binarios + Dialog como Centro (D5.3). Las entidades binarias siguen bajo D5.3; Línea es la excepción documentada. Origen: Bloque 16.

- **D6.3 — Unicidad del `codigo` de Línea es POR CENTRO, no por tenant.** Las redes urbanas de centros distintos son independientes: dos centros pueden tener cada uno su línea "1"; dos "42A" en el mismo centro se rechazan. Implementación: `assertCodigoLineaUnico(db, centroId, codigo, excludeLineaId?)` en `refs.ts` (análogo a `assertCIFUnico` de B9) + índice compuesto `(lineas: centroId+codigo)`. En UPDATE se revalida solo si el `codigo` cambia, con `excludeLineaId` para que la línea no choque consigo misma. Origen: Bloque 16.

- **D6.4 — Las entidades operativas nuevas nacen con auditoría canónica (`creadoPor`/`creadoEn` D3.7 + `actualizadoPor`/`actualizadoEn` D4.1) y NO arrastran `fechaCreacion`.** Línea es la primera: al no tener documentos legados, corta limpio al esquema canónico de auditoría. Refina el patrón para futuras entidades operativas (Paradas, Frecuencias, Tipos de turno), que deben nacer igual. La deuda de las 9 entidades viejas que aún usan `fechaCreacion` se rastrea en `TODO[unificar-fechacreacion-creadoen]`. Origen: Bloque 16.

- **D6.5 — Toda query de LISTADO desde cliente (`getDocs` con `where`) para un rol NO-super_admin debe incluir `where('tenantId','==',claims.tenantId)` para satisfacer la regla `read` `sameTenant(resource.data.tenantId)`.** Razón: en operaciones `list`, Firestore evalúa la regla SIN los documentos (`resource.data` es undefined), así que exige que la query CONSTRIÑA `tenantId` para poder probar la pertenencia — y de paso bloquea lecturas cross-tenant (un jefe filtrando un `tenantId` ajeno → `permission-denied`). Las colecciones super_admin-only (Tenants/Centros/Usuarios) nunca toparon con esto porque `isSuperAdmin()` cortocircuita el `||` antes de tocar `resource.data`; Líneas (B17) es la primera listada por un jefe. NO separar `get`/`list` con un `list` token-based (`allow list: isJefeTrafico() && token.tenantId != null` NO constriñe la query → leak cross-tenant: el jefe podría filtrar un centro de otro tenant y leer sus líneas). El patrón seguro es **constreñir la query, no relajar la regla** (verificado empíricamente con el SDK de cliente en B17). Aplica a futuros listados de jefe (frontend de Conductores, Paradas, Frecuencias). Origen: Bloque 17.

- **D6.6 — La duración de un turno (`duracionMinutos`, `duracionEfectivaMinutos`) se DECLARA explícitamente en el payload; NUNCA se calcula en backend desde `horaInicio`/`horaFin`.** Razones: (a) en turnos partidos la duración efectiva ≠ `fin − inicio` (hay corte intermedio); (b) en turnos que cruzan medianoche (`horaFin < horaInicio`) la aritmética se complica; (c) la duración a efectos de convenio puede diferir de la aritmética de horas (tiempo de bocadillo computable o no, redondeos del convenio). El backend valida **coherencia** (duraciones > 0, `efectiva <= total`, tramos del partido dentro de `[horaInicio, horaFin]` cuando NO cruza medianoche) pero NO impone la aritmética. Los consumidores futuros (cuadrante, optimizador) **consumen la duración declarada, no la recalculan**. Nota relacionada (sin canónica propia): TipoTurno resolvió el "eje mezclado" igual que Línea (D6.2) — se eliminó `tipoFranja` (que solapaba franja-horaria con `esPartido`) en favor de `esNocturno` boolean ortogonal + `esPartido` boolean; y su `estado` usa el patrón binario **D5.3** (`activo`/`obsoleto`, sin estado intermedio tipo suspensión estacional, a diferencia del enum-3 de Línea). Origen: Bloque 18.

- **D6.7 — La verificación de UI se hace con tests E2E de Playwright (`apps/web/e2e/`), no con checklist manual.** Reemplaza el verify manual en navegador de B17/B19. Patrón canónico del andamiaje (B20):
  - **Login una vez** vía proyecto `setup` que guarda `storageState` con **`indexedDB:true`** (Firebase Auth Web SDK persiste la sesión en IndexedDB, no en cookies/localStorage); los specs declaran `dependencies:['setup']` + `storageState` y arrancan autenticados, sin re-loguear.
  - **`webServer` propio en puerto FIJO** (`--strictPort`), independiente del puerto variable del dev server del usuario; el usuario no se preocupa del puerto.
  - **Emuladores = prerequisito asumido corriendo**; `globalSetup` los verifica (falla con mensaje claro si no), **siembra de forma idempotente** y hace **warm-up de los callables** (POST a sus endpoints) para evitar el cold-start del Functions emulator que cuelga la primera mutación de forma flaky.
  - Tests que **mutan estado resetean en `beforeEach`** recreando SOLO los datos de la entidad, **sin tocar los UIDs de Auth** (recrearlos invalidaría el `storageState`). `workers:1` (BD del emulador compartida → serie evita carreras).
  - **Cada entidad nueva con UI añade su propio spec** (`<entidad>.spec.ts`) + helper de reset.
  - Override ESLint para `e2e/**` + `playwright.config.ts` (node globals); E2E fuera de `tsc -b` (Playwright transpila aparte).

  A partir de B20, **la verificación de UI es automatizada** (`npm run test:e2e` con emuladores corriendo), no manual. Origen: Bloque 20.

- **D6.8 — Validación de NO-SOLAPAMIENTO temporal de tramos horarios.** Cuando una entidad tiene tramos `[horaInicio, horaFin)` que no deben solaparse dentro de un mismo agrupador (ej. Frecuencia: misma `lineaId` + `tipoDia` + `sentido`), se valida con la regla de intervalo **MEDIO-ABIERTO**: dos tramos solapan sii `aInicio < bFin && bInicio < aFin` (así tramos **adyacentes** 06:00–10:00 y 10:00–14:00 conviven sin falso conflicto; solo solapan si se pisan estrictamente). Se valida en el **callable** (NO en reglas Firestore — requiere leer otras filas del agrupador), y rechaza con `failed-precondition`. `excludeId` para que el `actualizar*` no choque consigo mismo; solo cuentan las filas **activas** (`activa===true`). En Frecuencia, además, `sentido: 'ambos'` colisiona con cualquier sentido (cubre ida+vuelta); en UPDATE se re-valida con los valores **efectivos** (payload ?? doc). Origen: Bloque 23 (primera validación de solapamiento temporal del proyecto). Reusable para futuros tramos no-solapables (paradas con ventanas, asignaciones, etc.).

- **D6.9 — Entidad SINGLETON por padre.** Cuando una entidad tiene exactamente UNA instancia por su entidad padre (ej. `Convenio`: uno por centro), se modela con **`id` = id-del-padre** (`id=centroId`), un **único callable UPSERT** (`guardar*`, NO `crear*`/`actualizar*` separados) que hace **set-si-no-existe / update-si-existe preservando `creado*`** (D3.7 en create, D4.1 en update), **sin campo `estado`** (no hay entre qué elegir; no aplica D5.3 ni soft-delete), **sin validación de unicidad** (el id determinista la garantiza por construcción, no hace falta `assert*Unico`), y **lectura por get-by-id directo** desde cliente — lo que **escapa del bug D6.5** del `list` (en un `get`, `resource.data` SÍ está disponible, así que `sameTenant(resource.data.tenantId)` evalúa bien sin necesidad de constreñir query ni índice). El callable veta los campos server-managed en el validador y, en UPDATE, los inmutables que identifican al padre (ej. `tenantId`). Reglas patrón `/lineas` (jefe-scoped + veto inmutables). Reusable para futuros singletons por padre. Origen: Bloque 25.

- **D6.10 — Materialización selectiva de asignaciones (el descanso ordinario NO se materializa).** En el cuadrante, el día libre normal de un conductor NO se persiste como documento: **"ausencia de fila `Asignacion` = día libre"**. Solo se crean documentos `Asignacion` para lo que **OCUPA** al conductor: `turno`, `reserva_presencial`, `reserva_localizable`, `vacaciones`, `baja`. El valor `'libre'` del union `TipoAsignacion` queda en el modelo por compat pero los callables `crear*`/`actualizar*` lo **rechazan** con `invalid-argument` (mensaje específico). Razón: materializar descansos serían decenas de miles de documentos vacíos al mes (≈178 conductores × ~60 días de horizonte) sin información útil. Coherente con la "materialización selectiva de servicios" del §8 (solo se crea `servicios` cuando hay algo que registrar). **Implicación para todo consumidor** (frontend del cuadrante, "mi horario", optimizador): al leer un cuadrante, la ausencia de asignación de un conductor en una fecha se interpreta como día libre — no se busca ni se espera un documento. Origen: Bloque 26.

- **D6.11 — Testing unitario (vitest) para lógica pura.** La lógica pura no trivial (algoritmos sin I/O: cálculo de calendario, transformaciones de datos, validadores de dominio…) se testea con **vitest en `packages/shared`** (ficheros `src/**/*.test.ts`, `npm test -w @albius/shared`), NO solo con scripts `verify-*.mjs` (que prueban I/O end-to-end contra el emulador) ni con E2E. Primer caso: `resolverTipoDia` (B27, 10 tests de casos límite). Criterio: una función pura **crítica** merece tests unitarios de sus casos límite (aquí, porque alimenta la demanda del optimizador). Las funciones que **SÍ** tocan Firestore (callables) se siguen validando con `verify-*.mjs` contra emulador (su valor es probar el I/O real + reglas), y la UI con E2E Playwright (D6.7). vitest es la tercera pata: pura, rápida, determinista, sin emulador. Origen: Bloque 27.

- **D6.12 — Lógica de escritura reutilizable (`logic/` separado del handler).** Cuando una operación de escritura debe ser invocable tanto por un **callable de usuario** (con `request.auth`) como por **otro contexto sin usuario** (un orquestador de servicio, otro callable), la lógica de escritura se extrae a una **función pura** en `apps/functions/src/logic/` con firma `(db, params, actorId) → resultado`, y el callable queda como **envoltorio fino** que hace SOLO auth + anti-cross (`assertJefePuedeTocarCuadrante` se queda en el callable) y delega con `actorId = invocadorUid`. Los **guards de identidad de usuario NO bajan** a la función extraída; esta solo valida lo **estructural** (cuadrante editable, fecha-en-mes) y ejecuta la mecánica de Firestore (batches). El `actorId` se persiste como `creadoPor`/`actualizadoPor`; convención de **actor de servicio `actorId='optimizador'`** (paralelo a `bootstrap-cli`/`system-seed` de D3.7/§14.9). Esto **habilita la Opción C** del optimizador (B29 llama estas funciones sin ser un usuario autenticado). Primeras funciones: `escribirAsignacionesLote`/`limpiarAsignacionesCuadrante`/`regenerarAsignaciones` (`logic/asignaciones.ts`) + `actualizarCuadranteDoc` (`logic/cuadrante.ts`). NOTA: operaciones que superan los límites de Firestore (500 ops/batch, 500 docs/transacción) **no son atómicas** — se trocean en batches y se documenta el no-atomicidad + la estrategia de reintento (idempotente). Origen: Bloque 28.

## 14. Procedimiento de trabajo (emergente, validado tras Bloques 3-5)

Patrón iterativo que ha demostrado valor en los últimos sub-bloques:

  Diseño → OK del usuario → Aplicación → OK del usuario → Verificación → OK del usuario → Commit → OK del usuario

Reglas operativas observadas:

1. **Pre-commit checks obligatorios**: `npm run build` + `npm run lint` antes de cerrar cualquier commit. Build incluye `tsc -b` (validación de tipos). Origen: lección incorporada tras `e879854` (push con lint en rojo que requirió hotfix `6d76b68`).
2. **Verificación empírica antes de aplicar cambios sugeridos**. Patrón: cuando se ve algo aparentemente raro (línea duplicada, archivo extraño, proceso colgado), primero verificar con `grep` / `cat` / lectura del log; modificar solo tras confirmar la condición real. Lección consolidada: el "bug aparente" fue artefacto visual en 3 ocasiones distintas (commit C2 del 3.2.e, `target` en bootstrap, Vite "colgado" del Bloque 5).
3. **Mensajes de commit detallados** con trazabilidad de decisiones `D-N` y `DUDA-N`, hashes referenciados, justificación de cambios técnicos. El commit debe permitir a un lector futuro entender el contexto sin abrir archivos adicionales.
4. **Deuda explícita**: cuando se pospone una decisión, se anota inmediatamente como `TODO[nombre-descriptivo]` en §12 con contexto, qué resolvería y cuándo encaja.
5. **Separación de scopes para scripts**: `apps/functions/scripts/` para testing infrastructure (`verify-*` que validan callables, `seed-*` que crean datos de prueba); `scripts/` raíz para operaciones CLI standalone (`bootstrap-*` para bootstrap del sistema). Cada tipo tiene perfil de fail-safe distinto: producción (6 capas en bootstrap), emulator-only (hardcoded en seed).
6. **Working tree por commit**: stage explícito por archivo, nunca `git add -A` o `git add .`. Cada commit tiene propósito autocontenido.
7. **Limpieza de procesos al cerrar sesión**: TaskStop por ID + verificación manual de huérfanos. El proceso `java` del Firestore emulator NO recibe el SIGTERM de Node y queda activo en cierres anteriores (3.2.c, 3.2.d, 3.2.e, 4, 5). Matar con `Stop-Process -Force` confirmando con CommandLine que es el Firestore correcto.
8. **`--target` o equivalente sin default** para cualquier script CLI que pueda tocar producción. Forzar elección consciente del operador (capa 1 de fail-safe).
9. **Sistema de auditoría con `creadoPor` en 3 niveles**: callable (uid real de `request.auth.uid`), bootstrap CLI (`'bootstrap-cli'`), testing seed (`'seed-test-user'`). Si en producción aparece `'seed-test-user'`, es alarma temprana de datos contaminados de testing.
10. **Diagnóstico antes que kill**: cuando algo parece colgado, leer log antes de matar proceso. El primer paso de diagnóstico es siempre `tail` del log + `netstat`. Lección: Vite estuvo listo en 449ms; el poll script fallido buscaba string con escapes ANSI que nunca matcheaban.
11. **Verificación de UI vía E2E Playwright, no manual** (D6.7). La UI se verifica con `npm run test:e2e` (Claude Code lo ejecuta), no con checklist manual en navegador. **Prerequisito: emuladores corriendo** (`npm run emulate`) antes de lanzar los E2E; `globalSetup` falla con mensaje claro si no los encuentra. Origen: Bloque 20.
